**Stats:** +204 / -25 across 4 files

## What it does

Adds support for extracting multiple path parameters from a URL using tuple syntax. Before this PR, `Path<T>` could only extract a single `:param` segment. Now you can write:

```rust
#[get("/orgs/:org_id/repos/:repo_id")]
async fn get_repo(Path((org_id, repo_id)): Path<(u64, u64)>) -> String { ... }
```

Tuples of up to 5 parameters are supported.

---

## Overall verdict

**This is a solid, well-structured PR that fills a real gap.** The approach is sound, the macro fix is correct, tests cover the important cases, and it ships with docs and an example. There are a few things worth discussing before merging, outlined below.

---

## What's good

1. **The macro fix is the real gem here.** Changing `route_macro_core` from requiring `Pat::Ident` to accepting any pattern (via a temp variable + rebind) is a necessary fix that unblocks not just tuple paths but any destructuring pattern in handler args. This is independently valuable.

2. **The `impl_multiple_params!` macro is clean.** The trailing-comma tuple trick `($($T),+,)` is idiomatic Rust for single-element tuple disambiguation. The error messages include the parameter name, which is helpful.

3. **Good test coverage.** Tests for 2-param, 3-param, missing param, and parse failure cover the main paths.

4. **Docs and example are included.** The routing.md update and `multi_path_params.rs` example make the feature immediately discoverable.

---

## Concerns and suggestions

### 1. The `SingleParam` sealed trait is a breaking change

This is the biggest issue. The current `impl<T: FromStr + Send> FromRequestParts for Path<T>` is replaced with `impl<T: sealed::SingleParam> FromRequestParts for Path<T>`, and `SingleParam` is only implemented for a hardcoded list of 14 primitive types.

**What breaks:** Any user type that implements `FromStr` and was used as `Path<MyType>` will stop compiling. This is a common pattern (e.g., `Path<Uuid>`, `Path<Slug>`, custom ID newtypes).

**Why it was done:** To avoid coherence conflicts between the blanket `impl<T: FromStr>` and the tuple impls. Tuples like `(u64, u64)` don't implement `FromStr`, so there's no actual overlap, but Rust's coherence checker can't prove that a future blanket `impl FromStr for (A, B)` won't appear.

**Suggestions:**
- At minimum, add `uuid::Uuid` and common ecosystem types to the list (behind feature flags).
- Better: consider a different trait design. For example, a marker trait `PathParam` that users can derive or implement, with a blanket impl for common types. This way users can opt-in their custom types.
- Alternatively, use a newtype wrapper for tuples (e.g., `PathTuple<(T1, T2)>`) to avoid the coherence issue entirely, keeping `Path<T>` working with any `FromStr` type. This would be less ergonomic but fully backwards compatible.

### 2. Positional extraction is fragile

Parameters are extracted positionally by iterating `params.iter()`, ignoring the parameter names entirely. This means:
- If the router ever reorders params, extraction silently breaks.
- Users can't match params by name, only by position.
- There's no validation that the number of `:param` segments in the route matches the tuple arity — a mismatch is a runtime error, not a compile-time one.

This is the same approach Axum uses, so it's a reasonable design choice, but it's worth documenting the positional-binding contract explicitly. Consider adding a compile-time check in the proc macro that counts `:param` segments and compares to the tuple arity.

### 3. Missing edge case tests

- **No macro-level test for destructuring patterns.** The macro change (accepting arbitrary patterns via temp variable rebinding) is the core enabler for this feature, but there's no unit test in `rapina-macros/src/lib.rs` that verifies a tuple destructuring pattern like `Path((a, b)): Path<(u64, u64)>` expands correctly. The existing 35 macro unit tests all use simple `Pat::Ident` patterns. The feature works (CI proves it via the example), but a dedicated macro expansion test would catch regressions if `route_macro_core` is refactored again.
- **4 and 5 parameter tuples** — these are generated by the macro but have no tests.
- **Mixed types** — no test for e.g. `Path<(u64, String)>`.
- **Single-element tuple** — `Path<(u64,)>` isn't generated, which is fine, but should be documented as unsupported.
- **Empty params** — what happens with `Path<(u64,u64)>` on a route with zero `:param` segments?

### 4. The `SingleParam` list is missing `usize`, `isize`, and `char`

These implement `FromStr` and are reasonable path parameter types. `usize` in particular is commonly used for indices.

### 5. Minor: trailing comment style

```rust
impl_multiple_params!(T1, T2, T3, T4, T5); //5th parameter just in case
```

This comment doesn't add value and reads a bit casually for library code. Either remove it or replace with a doc comment explaining the upper bound choice.
