# Design: `rapina migrate` Run Commands (Issue #520)

**Date:** 2026-04-22  
**Branch:** fix/migrate-command-execution  
**Milestone:** 0.12.0

## Problem

`rapina migrate new <name>` scaffolds a migration file but there is no way to apply, rollback, or inspect migrations from the CLI. Users must either call `run_migrations::<Migrator>()` at app startup (races on multi-replica, no rollback) or hand-write a `src/bin/migrate.rs` boilerplate every project.

## Solution: Shell-out via user-owned binary

The `rapina-cli` binary cannot link against the user's `Migrator` type at runtime. The chosen approach is the same pattern as `sea-orm-cli`: scaffold a small `src/bin/rapina_migrate.rs` in the user's project that calls into `rapina::migration::run_cli::<Migrator>()`. The CLI shells out to `cargo run --bin rapina_migrate -- <subcommand>`.

## Architecture

```
rapina (lib)
└── migration.rs
    ├── run_pending<M>()    (existing)
    ├── rollback<M>()       (existing)
    ├── status<M>()         (existing)
    └── run_cli<M>()        NEW — parses argv, reads DATABASE_URL, dispatches

rapina-cli (binary)
└── commands/migrate.rs
    ├── new_migration()     (existing)
    ├── init_migrate_bin()  NEW — scaffolds src/bin/rapina_migrate.rs
    └── run_migrate_cmd()   NEW — shells out to cargo run --bin rapina_migrate

src/bin/rapina_migrate.rs   (scaffolded by init or rapina new)
    fn main() {
        rapina::migration::run_cli::<migrations::Migrator>()
    }
```

`src/bin/` is auto-discovered by Cargo — no `[[bin]]` entry needed in `Cargo.toml`.

## `rapina::migration::run_cli<M>()`

Async function. Reads `std::env::args()` for the subcommand:

| Arg | Action |
|-----|--------|
| `up` | `M::up(conn, None)` — apply all pending |
| `down [--steps N]` | `M::down(conn, Some(N))` — default N=1 |
| `status` | `M::status(conn)` — print migration table |
| `fresh` | `M::fresh(conn)` — drop all tables + re-run all up |
| `reset` | `M::refresh(conn)` — down all + up all |

Reads `DATABASE_URL` from env (dotenvy loaded). Prints success/error, calls `std::process::exit(1)` on error.

## New CLI Subcommands

```
MigrateCommands:
  New { name }              (existing)
  Init                      NEW: writes src/bin/rapina_migrate.rs
  Up                        NEW: cargo run -q --bin rapina_migrate -- up
  Down { steps: Option<u32> }  NEW: cargo run ... -- down [--steps N]
  Status                    NEW: cargo run ... -- status
  Fresh                     NEW: cargo run ... -- fresh
  Reset                     NEW: cargo run ... -- reset
```

If `src/bin/rapina_migrate.rs` does not exist, `Up/Down/Status/Fresh/Reset` print:

```
Error: migrate binary not found.
Run 'rapina migrate init' to set up the migrate binary for this project.
```

## `rapina migrate init`

Creates `src/bin/rapina_migrate.rs` using `#[path]` so sub-modules resolve correctly (see module resolution section below).

**Note:** If `src/migrations/` does not exist yet, `init` still creates the bin file and warns the user to run `rapina migrate new <name>` first.

### Module resolution

`src/bin/rapina_migrate.rs` is a separate compilation unit and cannot access `src/migrations/` via a plain `mod migrations`. The fix: use `#[path]` pointing to the actual file. Per the Rust reference, inner module searches are resolved relative to the referenced file's directory — so sub-modules declared in `src/migrations/mod.rs` correctly resolve to `src/migrations/*.rs`.

```rust
// src/bin/rapina_migrate.rs — generated by `rapina migrate init`
#[path = "../migrations/mod.rs"]
mod migrations;

#[tokio::main]
async fn main() {
    rapina::migration::run_cli::<migrations::Migrator>().await;
}
```

No `src/lib.rs` required. No `[[bin]]` entry in `Cargo.toml` required.

## `rapina new` Integration

Templates that include a DB:

| Template | DB present | Scaffolds bin |
|----------|-----------|---------------|
| `rest-api` | no `--db` | no |
| `rest-api` | `--db` given | yes |
| `crud` | always | yes |
| `auth` | no `--db` | no |
| `auth` | `--db` given | yes |

## Startup `run_migrations` (existing)

No change. Remains available for single-node convenience. Not called by default in new templates. Issue notes it is a footgun on multi-replica — a doc note is sufficient.

## Tests

### Unit (rapina-cli)
- `init_migrate_bin()` writes correct file content
- `init_migrate_bin()` is idempotent (file already exists → error or skip)
- `run_migrate_cmd()` returns error with helpful message when bin file absent
- Arg parsing in `run_cli`: `up`, `down`, `down --steps 3`, `status`, `fresh`, `reset`

### Integration (rapina lib, `#[cfg(feature = "sqlite")]`)
- `run_cli` with `up` applies migrations (SQLite in-memory via `DATABASE_URL=sqlite::memory:`)
- `run_cli` with `down --steps 1` rolls back
- `run_cli` with `status` does not error
- `run_cli` with `fresh` drops + re-runs
- `run_cli` with `reset` refresh cycle

Build on existing `tests/migration_test.rs` pattern.
